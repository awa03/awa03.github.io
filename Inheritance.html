Inheritance

Classes can be linked together in order to pass data through one another. This idea is known as Inheritance This base class / derived class relationship is often referred to as an "is a" relationship because the derived object really is an instance of the base object -- it's usually just a bit more specific (like a subcategory). The take out common features within the base class if they are used within the inherited classes It is essentially just a subcatagory of the base class. Note that this is different from the "has-a" relationship, in which objects are embedded inside of others (i.e. one object is a component of another).

Syntax: class derivedClassName : public baseClassName<>

Protection Levels:

Public signifies in this case that it is publicly derived from the base class name In order to move data between these classes through inheritance we use protections, and public member data. Protection levels include public and private;

public - Any member that is public can be accessed by name from anywhere.
public - Any member that is private can be accessed directly only by the class in which it is declared.

The keyword protected: allows for inherited classes to have access to those member functions/data. apart from this, it operates the same as the private member data.
protected - Any member that is protected can be accessed directly by the class in which it is declared, and by any classes that are derived from that class (but not from anywhere else).

Constructors in derived classes:

In this case, the derived object "is-an" instance of the base class as well. So, when a derived object is created, the constructors from the base and derived classes will run. The order in which they run is important, too -- the base class constructor runs first, then the derived. "In other words the car class would run before the honda class"

Since these classes are derived from one another they inherit from the base class. Take for example a students grad a class representing the students as a wholes data would be inherited by all the students, however the subclass computer science majors may only be inherited by a few students. However, if the called function should look different in each of the subclasses (an example being a grade calculation function; different grading styles may occur per major) we need to override the function.We just need to redeclare the "GradeReport" within the sub class functions. Virtual Functions

We cannot use a basic loop (as given in the class example) to store the list of students. This is because of possible seperation within the array- some students may be undergraduates, some may be graduates, etc. To account for this we can use a few solutions. We may create arrays within each subtype, however this is not scalable so may be hard to use in larger applications. We can get around this issue by using a base class pointer. Normally a pointer may only point at one type however, a pointer to a base class can be pointed at an object derived from the base class. Similarly, a base class type reference variable can refer to an object derived from that base class.

Student s; Grad g; Undergrad u;
Student *sp1, *sp2, *sp3;
sp1 = &s; // sp1 is now pointing at Student object
sp2 = &g; // sp2 is now pointing at Grad object
sp3 = &u; // sp3 is now pointing at Undergrad object
Herogeneous List

A heterogeneous list is made by creating a single array of pointers to the base class. The pointers can be pointed to any objects which are derived from the base class. So it is essentially a list of pointers to class object, an array of objects. This allows us to store diffrent data types within the array because they are all objects. Everything is a pointer to the base class

Student * list[300000]; // an array of Student Pointers
list[0] = &g; // using the earlier grad object
list[1] = &u; // undergrad object
list[2] = new Grad; // dynamic grad object
// can add any subtype of student
Virtual Functions

Heterogeneous Functions may solve the first issue, however who's version of Grade Report will run? if we attempted to run functions within the student list (like list[i]-> GradeReport();) we run into the issue of binding because list[i] is declared as a student pointer. The function call is bound to the definition. This means that the definition is static and the compiler cannot know what we would be pointing towards, because of this we need a way to bind to the correct definition at run time (dynamic).
The virtual keyword allows us to link the definition dynamically. This will override the base class function only when it is called through a pointer, so it will run the target version. The syntax (in the student example) can be seen below

class Student
{ public:
virtual void GradeReport();
};

Now the grade report function can be called through a pointer base class. Virtual functions are not specific to heterogeneous lists, they just work well in conjunction.

Student *sp1, *sp2, *sp3;
Student s; Grad g; Undergrad u;
sp1 = &s;
sp2 = &g;
sp3 = &u;
sp1->GradeReport(); // runs Student's version
sp2->GradeReport(); // runs Grad's version
sp3->GradeReport(); // runs Undergrad's version
Abstract Classes

In some situations the base class may be so obscure that we may not want to build objects out of it. An Example being things on earth. If we are referencing dogs, things on earth may be too loose of a basis to transfer any direct meaningful inheritance to the dog object. Or the course example: For instance, class Student might not have enough information for the full grade reports -- we need to know what kind of student (i.e. which subtype). If you do not want to actually do anything in the Student's version of GradeReport, you are allowed to leave out its function definition altogether.

virtual void GradeReport()=0;

The function shown above is giving its own definition (known as a pure virtual function), this does not have to be the case, the function may be left without an "=?", but it is often dont when the base class version has essentially no work to do.

Any class with one pure virtual function is known as an abstract class. An abstract class cannot be instantiated (cannot build an object of this class type). They are used as base classes for inheritance heiarchies , and they are intended only as a place to put data and functions common to classes derived from them, as well as a place for virtual functions. Abstract classes can still be used to build pointers, though, which is useful when taking advantage of virtual functions.

Shape s; // illegal declaration, since Shape is an abstract class
Shape * sptr; // legal declaration. sptr may point to derived objects
